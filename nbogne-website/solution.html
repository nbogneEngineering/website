Architecture

The Data Journey

What happens when EMR A needs to send a lab result to EMR B.

Step 1

EMR Sends Data

Source EMR sends patient data via API. Data arrives in whatever format the EMR supports‚ÄîHL7v2, FHIR, proprietary JSON.

Step 2

nBogne Receives

Data loaded into RAM. Never touches disk. Clock starts on 60-second maximum processing window.

Step 3

Format Transformation

Our format library identifies source format and transforms to destination's expected format. HL7v2 becomes FHIR. CSV becomes JSON.

Step 4

Compression & Routing

Z-Standard compression reduces payload 60-80%. Routing rules determine destination. Compressed data transmitted.

Step 5

Delivery Confirmation

Destination EMR confirms receipt. nBogne logs metadata: transaction ID, timestamp, source, destination, size, status, hash.

Step 6

Data Purge

Clinical data cleared from RAM. Total time: typically under 10 seconds. Maximum allowed: 60 seconds. No trace remains.

Security Model

Security Through Architecture

Traditional security asks: "How do we protect stored data?" We ask: "What if there's nothing to protect?"

No Data at Rest

Clinical data never touches disk. RAM-only processing means even physical server access yields nothing.

No Honeypot

Centralized health data repositories are high-value targets. nBogne has no repository to target.

Cryptographic Integrity

Every transaction includes a SHA-256 hash proving content wasn't altered‚Äîwithout revealing content.

Breach = Metadata Only

Worst-case breach exposes transaction logs: when data moved, between whom, how large. Never what was in it.

Vendor Exit Safety

If nBogne shuts down tomorrow, no patient data is affected. We can't lose what we don't have.

Compliance by Default

Ghana DPA (zero retention), HIPAA (ATNA audit), GDPR (nothing to erase). Architecture handles compliance.
